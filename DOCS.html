1 · Arquitectura

Tres piezas: un cliente Expo universal (nativo + web), un backend Supabase donde vive toda la lógica sensible, y R2 para las imágenes.

Cliente (client/)

  • Expo SDK 56 + expo-router con grupos de rutas: (auth), (tabs), album, sticker, join, pack, trade, admin.
  • React Query para data fetching; hooks por dominio en src/lib/queries/ (albums, stickers, packs, trades, daily, presets, profile, uploads, admin).
  • Design tokens en src/constants/theme.ts (fuentes Anton / Manrope / SpaceMono, paleta papel + tinta).
  • Web: web.output = "single" (AsyncStorage no funciona en SSR). Desktop capea el ancho del contenido, no el ScrollView.

Backend (supabase/)

  • ~55 migraciones con schema + RLS + RPCs SECURITY DEFINER. Toda mutación de negocio pasa por una RPC fn_* que valida ownership, estado y límites del lado del server.
  • 6 Edge Functions (Deno): open_pack, redeem_qr, revenuecat_webhook, upload_image, generate_qr, upload_preset_image. Helpers compartidos en _shared/http.ts (CORS, auth, respuestas) y _shared/r2.ts (R2 vía aws4fetch).
  • pg_cron activo para el sobre diario y limpiezas.
  • Proyecto: baexxbixcrhngbjptlkt (sa-east-1). PostgREST con db_max_rows = 2000 (el álbum especial supera las 1000 filas).

Storage (Cloudflare R2)

  • Bucket mi-album-figuritas con dominio público r2.dev.
  • La DB guarda keys, nunca URLs; el cliente arma la URL con r2Url(key). Cada imagen se sube en dos tamaños (thumb + large).

Auth

  • Magic link de Supabase. Deep link mialbum:// parseado por useDeepLinkAuth (token en fragment) y useJoinDeepLink para mialbum://join/<CODE>.
  • Las Edge Functions reenvían el JWT del caller para que auth.uid() funcione dentro de las RPCs.

3 · Flujo del owner

  1. Crear álbum (nombre, cantidad de figuritas, carátula y sobre — imagen propia o preset del admin).
  2. Cargar figuritas: individual (con número, nombre y rareza) o carga masiva (varias imágenes → números libres en orden, nombre #NN, rareza común). Modo reordenar con swap atómico (fn_swap_sticker_positions).
  3. Configurar economía: sobre diario (diario/semanal) y/o QR PRO, tamaño del sobre (1–10).
  4. Editar hojas: color, textura, composición, títulos (ver §7).
  5. Publicar: valida numeración completa server-side. Publicado = inmutable e irreversible (regla del mundo físico). Después de publicar solo se pueden editar las hojas.
  6. Compartir: código de 6 caracteres (tap para copiar), link https clickeable (mi-album.pages.dev/join/CODE — los custom schemes no se linkifican en WhatsApp; helper joinLinkFor()), o póster QR.
  7. Ciclo de vida: archivar (reversible) · retirar (si hay jugadores: read-only, conservan colección) · eliminar (sin jugadores, confirmación con email). El owner no juega su propio álbum.

4 · Flujo del jugador

  1. Unirse por código, deep link o botón en álbumes públicos.
  2. Conseguir figuritas: sobre periódico, QR del owner, o intercambios.
  3. Abrir sobres (pack/open): animación con sacudida, reveal escalonado, ribbons NUEVA/REPE y SFX sintéticos (lib/sfx.ts — respeta el switch de silencio, degrada sin el módulo nativo). Tap en una carta revelada → vista grande de la figurita. El álbum de dinos tiene sonidos temáticos (rugido al abrir, chillido en épica/legendaria; mapa albumSfxTheme() extensible a otros álbumes).
  4. Pegar: las recibidas entran pasted=false (borde gold, badge PEGAR); tap para pegar en la grilla.
  5. Completar: al pegar todas se corta el sobre diario automáticamente. También se puede silenciar por álbum (daily_muted).

El tab Sobres es el tablero: badge rojo con reclamables + sobres sin abrir, orden reclamables primero. El sorteo de figuritas ocurre server-side en open_pack según rareza.

Sobres por publicidad (rewarded ads · AdMob)

Migración 0059_ad_packs.sql. El jugador mira un video con recompensa y recibe un sobre extra. Solo Android: el SDK de AdMob es nativo (el rewarded web de Google está en beta, limitado a juegos H5) y iOS está descartado. En web/iOS el botón no existe (ADS_SUPPORTED = false).

  • Elegibilidad: solo los 3 álbumes especiales (avatares 29a1fa90…, 0..1000 ecbf4497…, dinos d1227449…), publicados, siendo miembro.
  • Tope server-side: 2 sobres/día por usuario, global entre los 3 álbumes, día en hora argentina. fn_claim_ad_pack lo valida SIEMPRE — llamar la RPC sin mirar el ad no otorga más que esos mismos 2 sobres, así que no hay economía que romper. Errores: P0210 ad_limit_reached, P0211 ad_not_available.
  • Estado para la UI: fn_ad_pack_status → {enabled, used, remaining, limit}; hook useAdPackStatus.
  • Cliente: wrapper por plataforma lib/rewarded-ad.android.ts (real: init lazy, load con timeout 12s, resuelve solo con EARNED_REWARD) y lib/rewarded-ad.ts (stub). Botón en la vista del jugador; agotado el cupo muestra "volvé mañana (2/2)". El sobre entra como pack_source = 'ad' y se abre como cualquier otro.
  • Ids actuales: App ID y ad unit de TEST de Google (sirven para desarrollar sin cuenta AdMob; en __DEV__ el wrapper fuerza test ads siempre).

Paso a paso para pasar a publicidades REALES (producción)

  1. Publicar la app en Play Store (aunque sea closed testing → producción): AdMob exige verificar la app contra su ficha de la tienda para servir ads reales.
  2. Crear cuenta AdMob (admob.google.com, con la cuenta Google del negocio) y completar los datos de pago (paga a partir de ~USD 100 acumulados).
  3. En AdMob: Apps → Agregar app → Android → vincular con el package com.mialbum.figuritas de Play Store. Copiar el App ID (formato ca-app-pub-XXXX~YYYY, con tilde).
  4. Crear un bloque de anuncios Rewarded dentro de esa app. Copiar el Ad Unit ID (formato ca-app-pub-XXXX/ZZZZ, con barra). Son dos ids distintos — no confundirlos.
  5. Reemplazar en el código (los dos puntos están marcados con TODO(admob)):
    • client/app.json → clave raíz react-native-google-mobile-ads.android_app_id (el de la tilde).
    • client/src/lib/rewarded-ad.android.ts → REAL_AD_UNIT (el de la barra).
  6. Rebuild de producción: eas build -p android --profile production (el App ID va al manifest nativo — un cambio de app.json no llega por OTA).
  7. app-ads.txt: si la ficha de Play declara un sitio web del developer, publicar en ese dominio el archivo app-ads.txt que AdMob te da (Apps → Ver detalles). Sin esto AdMob puede limitar el fill.
  8. Consentimiento (si hay usuarios UE): hoy pedimos requestNonPersonalizedAdsOnly; para cumplir GDPR formalmente hay que integrar el UMP SDK (viene en el mismo paquete: AdsConsent) y mostrar el formulario de consentimiento de Google antes del primer ad. Para lanzamiento solo-Argentina se puede posponer.
  9. NUNCA tocar ads reales en dispositivos propios durante pruebas (riesgo de ban por invalid traffic). Para probar en release: registrar el dispositivo como test device en AdMob, o seguir usando builds de desarrollo (fuerzan test ads solos).
  10. Verificación final: instalar el build de producción en un test device registrado → mirar el ad → confirmar que llega el sobre y que el tope de 2/día corta.

6 · Intercambios

  • Matches por álbum: fn_album_matches cruza tus faltantes con repes de otros (cap de 5 pares por contraparte). "Para cambiar" incluye repes reales (badge REPE) y sin pegar (badge PEGAR).
  • Ofertas 1:1: elegís qué das y qué recibís; la contraparte acepta o rechaza. Push + badge en el tab Cambios.
  • Reglas del owner: intercambios on/off y límite por día/semana/mes. El límite se chequea al aceptar; fn_trade_limit_status alimenta el banner informativo del jugador.

Todo vive en client/src/lib/page-config.ts (data) + edit-pages-modal.tsx (UI) + album-pager.tsx (render) + page-texture.tsx (SVG). Modelo: defaults a nivel álbum + overrides por hoja, persistidos como keys cortas — el cliente resuelve hex/dimensiones, así se pueden retocar sin migración.

Sin migraciones para extender: fn_update_album_pages guarda page_overrides como jsonb pasante (solo valida que sea array). Campos nuevos del override (título, colores, alineaciones) viajan adentro sin tocar el backend.

Cómo se dibuja una hoja

  • La hoja tiene tamaño fijo (calculado del layout 3×4). El fit achica las celdas si la composición/título no entra — nada desborda nunca.
  • El alto del título se mide (onLayout) porque puede envolver a 2 renglones; la grilla se recalcula.
  • Columna con espaciadores flex:1: la grilla queda centrada y el título se ancla al borde o pegado a la grilla.
  • Texturas: <Pattern> SVG con tile chico + rect que llena el viewport. En web necesitan px exactos (medidos con onLayout) e ids únicos por textura.
  • El owner tiene un botón de lápiz en cada hoja del pager que abre el modal directo en el editor de esa hoja.

El modal de hojas (3 pantallas)

  1. Ajustes: defaults del álbum + card "Personalizar hojas".
  2. Grilla de hojas: previews en 3 columnas (FlatList virtualizada), punto dorado = personalizada.
  3. Editor de hoja: overrides individuales, navegación entre hojas con flechas.

8 · Pipeline de imágenes

El procesamiento es 100% client-side (expo-image-manipulator); la Edge Function upload_image es un proxy trivial a R2. Razón: sharp en Deno Edge tiraba WORKER_RESOURCE_LIMIT.

KindThumbLargeCalidad JPEG
sticker512px1600pxthumb 0.85 · large 0.92
cover512px1200px
pack512px1200px
avatar512px1024px
  • Sin doble compresión: el ImagePicker usa quality: 1; la única compresión JPEG la hace el manipulator.
  • Sin upscaling: si el original es más chico que el target, se re-encodea sin resize.
  • Celdas adaptativas: StickerCell se mide y cambia de thumb a large cuando supera los px físicos del thumb (~550, contando PixelRatio) — clave para fichas con texto en layouts grandes.
  • Las imágenes ya subidas conservan su versión vieja hasta re-subirse.
  • R2 nunca borra: las keys huérfanas (por reemplazo o delete) se acumulan a costo despreciable — decisión consciente, hay plan de GC si algún día molesta.

9 · Avatares desbloqueables

  • Los users no suben fotos (moderación): eligen de la galería curada por admin (presets kind avatar).
  • Gimmick: existe un álbum público de avatares; preset_images.sort_order = número de figurita. Pegás la figurita N → desbloqueás el avatar N. Libres para todos: {1, 4, 20, 22}.
  • Gate real server-side en fn_update_avatar; estado para la UI en fn_my_avatar_unlocks. El picker muestra los bloqueados sin spoiler (candado + número).

10 · Convenciones y decisiones técnicas

Backend

  • Mutaciones solo vía RPC SECURITY DEFINER con checks explícitos (owner, admin, estado). Errores de negocio con errcode Pxxxx → copy en lib/errors.ts.
  • RPC batch para evitar N+1: listas que necesitan data por item reciben uuid[] y devuelven todo junto (fn_album_progress, fn_my_daily_status, bundles fn_home_bundle…).
  • Side-effects no esenciales (push) van con exception when others then null — nunca voltean la transacción del caller.
  • plpgsql no valida SQL al crear, solo al ejecutar: probar cada RPC nueva en runtime.
  • Si un campo altera un invariante (ej. number_start), grepear todas las RPCs que lo asumen.
  • Listados sin .range() quedan capados por max_rows (hoy 2000) — paginar si algo pudiera superarlo.
  • pgcrypto vive en schema extensions: funciones que lo usan llevan set search_path = public, extensions.

Cliente / Web

Regla de oro web: Alert.alert de react-native es no-op en react-native-web. Importar siempre { Alert } de @/lib/alert (shim: en web usa window.alert/window.confirm). Para confirms ricos, BottomSheet in-app.
  • Modales nuevos: sobre BottomSheet (base común con backdrop, footer sticky, avoidKeyboard). ScrollView interno con flexShrink: 1 — nunca maxHeight: '100%' (rompe el scroll en web).
  • Edge Functions desde el cliente: siempre vía callEdgeFunction() (token, headers, timeout, parseo de errores).
  • Gestos + scroll en web: todo GestureDetector que conviva con scroll vertical lleva touchAction="pan-y" (RNGH pone touch-action: none y mata el scroll táctil).
  • Medir contenedores con onLayout, no useWindowDimensions, cuando hay caps de ancho (desktop). SVG en web necesita width/height en px explícitos.
  • Cap desktop sobre el contenido: contentContainerStyle=[base, useDesktopCap(n)]; envolver el ScrollView deja la barra de scroll flotando.
  • Queries: toda lectura hace if (error) throw toAppError(error); pull-to-refresh usa isRefetching.
  • State global compartido vía Context (ProfileProvider); useNow() singleton para countdowns.
  • Keys cortas en DB (colores, layouts, presets) — el cliente resuelve valores; cambiar un hex no requiere migración.

11 · Operativas / comandos

# Typecheck del cliente
cd client && npx tsc --noEmit

# Aplicar migraciones al proyecto linkeado
supabase db push

# Deploy de una Edge Function (tras cualquier cambio)
supabase functions deploy <name>

# Regenerar tipos tras migraciones (OJO: Out-File utf8, NO '>' que mete BOM UTF-16)
supabase gen types typescript --linked | Out-File -Encoding utf8 client/src/lib/database.types.ts

# Regenerar efectos de sonido
node client/scripts/gen-sfx.js
  • Si una migración cambia la firma de una RPC granteada: drop function + create + re-grant.
  • Módulos nativos nuevos (expo-camera, expo-clipboard, expo-audio…) requieren rebuild del dev build de EAS; en web funcionan sin rebuild.
  • Para probar trades/joins hacen falta 2 cuentas (cerrar sesión y registrarse con otro mail).
  • Álbumes especiales protegidos contra borrado (P0201, migración 0058): avatares, 0..1000 (number_start = 0), dinosaurios, más un id histórico sin identificar. Espejo client-side en PROTECTED_ALBUM_IDS.
  • Estadísticas: Panel admin → Estadísticas (fn_admin_stats, migración 0057): usuarios, activos/día y logins/día (desde auth.audit_log_entries, hora argentina), álbumes, figuritas, sobres, cambios.
